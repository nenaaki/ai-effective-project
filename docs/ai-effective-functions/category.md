### spec駆動での設計のポイント

以下の要素を明示できる場合、生成されるコードの精度が向上します。

1.  **データモデル（型）**: 「何を表示/保存するのか」を定義したクラスやInterface。
2.  **インターフェース（口）**: 「どうやって呼び出すのか」を定義したAPI定義やMethodシグネチャ。
3.  **ステート（状態）**: 「どんな状態がどのように遷移するのか」を定義したマトリクスや図(MermaidやD2)。
4.  **制約条件（ルール）**: 「何をしてはいけないか」というバリデーションや権限ルール、例外条件。


## AIへの指示難易度が高い実装カテゴリ

| カテゴリ | 具体的な機能例 | 指示が難しい理由（ボトルネック） |
| :--- | :--- | :--- |
| **横断的・複合的な整合性** | 複数画面にまたがるリアルタイム同期<br />複雑な排他制御（楽観ロック・悲観ロック） | 影響範囲が「ファイル単位」を超え、システム全体のライフサイクルや通信のタイミングに依存するため。 |
| **恣意的な例外ルールの塊** | 業界固有の複雑な商慣習<br />法改正に伴う特例措置<br />複雑な税計算（日本特有の控除ルール等） | 処理に論理的な一貫性がなく、「パターンの推論」が効かない。<br />全ての条件を仕様書に書き下す労力が実装より重くなる。 |
| **高度な対話型UI・操作感** | ドラッグ＆ドロップによる複雑な並び替え<br />キャンバス上での描画<br />スクロール連動アニメーション | 「座標」「速度」「慣性」といった動的な数値を言葉で定義するのが困難。<br />1ピクセルのズレや数ミリ秒の違和感を伝えにくい。<br />視覚効果をAIは理解しないことが多い。 |
| **不可逆・破壊的な一括操作** | 大規模なデータマイグレーション<br />外部サービスとの複雑な一括連携<br />自動マージ処理 | 「失敗した時のロールバック戦略」や「例外データのパターン」が無限に存在し、安全性を担保するための制約指示が膨大になる。 |
| **ヒューリスティックな探索** | 曖昧な検索ワードの意図解釈<br />類似度の重み付け調整<br />推薦アルゴリズムのチューニング | 「正解」の定義がユーザーの主観やデータ分布に依存するため、プロンプトで「ちょうど良い塩梅」を指定できない。 |

### 指示難易度が高いものの技術的背景

1.  **コンテキストの断片化の壁**:
    AIは一度に参照できる「文脈」に限界があります。システム全体に影響する機能（認証・認可の伝播など）は、個別のファイル生成だけでは「辻褄」が合いにくくなります。
2.  **暗黙的な前提条件の壁**:
    「当然こう動くべき」という、人間同士なら共有できている「Webアプリの標準的なお作法」や「OSの挙動」を、AIに対しては全て明示的に言語化（プロンプト化）する必要があります。
3.  **フィードバックループの物理的距離**:
    「触ってみて微調整する」という作業は、現状のAI（コード生成→デプロイ→確認）ではループが長く、人間が直接CSSやロジックを1行書き換える方が圧倒的に早い領域です。

### 指示難易度を下げる手段

これらをあえてAIに実装させる場合、**「自然言語での説明」を諦める** のが近道です。（目安： **1画面内に収まる程度の文章で仕様を明文化できるか**）

* **図解のコード化**: 複雑な遷移は言葉ではなく、MermaidやD2のコードとして「構造」を直接食わせる。
* **サンプルデータによる定義**: ルールを説明する代わりに「入力Aなら出力B」「入力Cなら出力D」というテストケース（入出力の対）を大量に与えてパターン認識させる。
* **ボイラープレートの分離**: 難しいロジック（核）は人間が書き、その周りの「データの口（I/O）」や「表示パーツ」だけをAIに任せるという役割分担を明確にする。
